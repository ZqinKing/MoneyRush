# GNU AGPL v3 与商业授权说明（中文解读）

> 本文件是对仓库根目录 [`LICENSE`](./LICENSE) 的中文说明，便于中文读者快速理解使用边界。  
> **具有法律效力的正式文本仍以英文版 `LICENSE` 为准。**

## 当前许可模式

本项目现在采用的是：

- **社区版许可证：GNU Affero General Public License v3.0（AGPLv3）**
- **商业授权：由版权所有者单独提供**

这是一种常见的**双重许可 / 双轨授权**思路：

- 如果你愿意遵守 AGPLv3，就可以按 AGPLv3 使用、修改、分发本项目；
- 如果你不希望遵守 AGPLv3 带来的开源义务，就应当另行联系版权所有者获取商业授权。

## 这是不是开源许可

是。AGPLv3 是一份**正式、成熟、标准化的开源许可**，属于 GNU / FSF 的 copyleft 许可证体系。

但它不是宽松开源许可。它比 MIT / Apache 之类更强，尤其适用于服务端和网络交互场景。

## 个人用户可以怎么用

个人开发者、研究者、学习者、爱好者，通常可以：

- 阅读源码；
- 本地运行和研究；
- 修改源码；
- 在遵守 AGPLv3 的前提下分发或部署；
- 将修改后的版本继续按 AGPLv3 开源出去。

## 商业公司能不能用

**能不能用，要分两种情况看：**

### 情况 A：愿意遵守 AGPLv3

如果商业主体愿意按 AGPLv3 的要求履行义务，那么它并不是“绝对不能商用”。

也就是说：

- AGPLv3 **不是**“商业使用绝对禁止”许可证；
- 它是“商业使用可以，但要接受强 copyleft 义务”的许可证。

### 情况 B：不愿意遵守 AGPLv3

这通常就是商业授权存在的意义。

例如，某个公司：

- 不想公开其修改版本源码；
- 不想承担 AGPLv3 在网络服务场景下的开源义务；
- 想把本项目集成到闭源商用产品、商用 SaaS 或专有内部系统中；

那么更稳妥的路径就是：**联系版权所有者获取单独商业授权。**

## AGPLv3 最关键的影响是什么

对这个项目最重要的是：

- 如果你**修改了** AGPL 覆盖的程序；
- 并且把这个**修改后的版本**作为网络服务提供给用户使用；
- 那么 AGPLv3 通常要求你向这些用户提供该修改版本的对应源码获取方式。

这也是 AGPLv3 比普通 GPL 更“适合服务端项目”的原因。

## 一个常见误区

很多人会把 AGPL 理解成：

> “只要调用它的 API，整个系统都必须全部开源。”

这个说法**过于绝对**。

更准确的理解是：

- AGPLv3 重点约束的是 **AGPL 程序本身及其修改版本**；
- 是否进一步影响外围系统，要看耦合方式、组合方式以及是否构成同一衍生作品；
- 不是一句“碰到了 API 就整个系统全开源”能概括的。

因此，商业团队通常会因为 AGPL 的合规风险而选择购买商业授权，而不是在边界上做激进解释。

## 如果你想商用，应该怎么办

如果你的目标是：

- 闭源部署；
- 不公开修改版本；
- 把本项目作为商用 SaaS / 商用产品 / 客户交付的一部分；
- 获得更宽松的专有使用权；

那你应优先查看仓库中的 [`COMMERCIAL-LICENSE.md`](./COMMERCIAL-LICENSE.md)，并通过维护者公布的联系方式申请商业授权。

## 简短结论

可以把当前许可模式简单理解成：

- **愿意遵守 AGPLv3：可以使用社区版**
- **不想遵守 AGPLv3：去申请商业授权**
